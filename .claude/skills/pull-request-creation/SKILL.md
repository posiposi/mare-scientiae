# Pull Request Creation Workflow

開発フロー第4段階「PR作成」を実行する。実装・コミット済みの作業ブランチを push し、日本語のタイトル・本文でプルリクエストを作成する。

## 対象

$ARGUMENTS

## 前提

- 対象 Issue に紐づく実装が完了している
- 必要な変更がすべてコミット済みである
- 作業ブランチが `main` ではない
- `api` コンテナ内で `gofmt -w .` と `go vet ./...` が通っている

未充足の場合は実装フェーズに戻るよう案内する。特に未コミットの変更が残っている場合は `/commit-commands:commit` での分割コミットを先に促す（CLAUDE.md「コミットの粒度は最低限の機能単位または修正項目一つずつ」）。

## 制約

- **git push および PR 作成の実行前に、必ずユーザー承認を取得する**。承認には「承認」「OK」「進めて」等の明示的な肯定表現を要する
- **コミットメッセージ・PR タイトル・PR 本文はすべて日本語で作成する**。`/commit-commands:commit-push-pr` 呼び出し時にこの旨を明示指示する
- **スコープは PR 作成の完了まで**。レビュー実行（`/code-review:code-review`）・マージ・ブランチ削除は本スキルの範囲外とし、呼び出し元の責務に委ねる

## フェーズ

### 1. 事前確認

以下を確認し、未充足があればユーザーに通知して次に進まない。

- `git status` で未コミットの変更が残っていないか
- `git branch --show-current` が `main` 以外の作業ブランチか
- 対象 Issue 番号が特定できているか（PR 本文で参照するため）

作業ブランチ名・Issue 番号・コミット履歴を把握した上で次フェーズへ進む。

### 2. PR タイトル・本文ドラフトの作成

以下を読み取り、PR の差分と目的を整理する。

- `git log main..HEAD --oneline` でコミット一覧
- `git diff main...HEAD` で差分全体
- 対象 Issue の本文・コメント（`mcp__github` 経由）

これらをもとに日本語でドラフトを作成する。

**PR タイトル**: 簡潔な 1 行。Issue 番号は**含めない**。

例: `書籍一覧取得APIの実装`

**PR 本文テンプレート**:

```
## 概要
<このPRで達成したこと>

## 変更内容
- <変更点1>
- <変更点2>

## 関連Issue
Closes #<番号>

## 動作確認
- <確認方法1>
- <確認方法2>
```

### 3. ユーザー承認の取得

以下をユーザーに提示し、明示的な承認を得る。承認が得られるまで push も PR 作成も行わない。

- push 予定のブランチ名
- base ブランチ（通常 `main`）
- PR タイトル案（日本語）
- PR 本文案（日本語）

修正要求があればタイトル・本文を修正して再提示する。

### 4. `/commit-commands:commit-push-pr` の実行

承認後、`/commit-commands:commit-push-pr` を呼び出す。呼び出し時のプロンプトに以下を明示する。

- 「コミットメッセージ・PR タイトル・PR 本文はすべて**日本語**で作成する」
- 承認済みの PR タイトル案と本文案をそのまま使用すること
- base ブランチ（通常 `main`）

コマンド実行後、作成された PR の URL をユーザーに通知してスキル実行を終了する。

## 原則

- **承認を飛ばさない**: push・PR 作成は取り消しコストが高いため、ドラフト提示と承認を必ず経由する
- **日本語を徹底する**: 本プロジェクトは日本語運用のため、PR 関連の自然言語成果物はすべて日本語で記述する
- **スコープを守る**: レビュー・マージ・ブランチ削除に踏み込まず、PR 作成完了をもって終了する

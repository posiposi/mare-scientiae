# Design Workflow

開発フロー第2段階「設計」を実行する。Issueに基づいて必要な観点を動的に決定し、サブエージェントで並列設計、統合Planをユーザー承認のもと確定させた上で実装フェーズへ引き継ぐ。

## 対象

$ARGUMENTS

## 前提

- 対象のGitHub Issue番号が特定済みであること
- `main`から分岐した作業ブランチが作成済みであること

未満の場合は開発フロー第1段階「Issue取得」に戻るよう案内する。

## 制約

- **Plan modeを使用する**。設計フェーズ中はコードベースへの変更を一切行わない
- **ユーザー承認前に実装フェーズに進まない**。承認には「承認」「OK」「この方針で進めて」等の明示的な肯定表現を要する
- **推測で設計しない**。Issueが曖昧な場合はユーザーに質問し、既存コードの把握が不十分な場合は`code-analyzer`サブエージェントで調査してから設計に入る

## フェーズ

### 1. Issue 内容の把握

`mcp__github`経由でIssue本文・コメント・関連Issueを取得し、以下を整理する。

- ゴール（何を達成するか）
- 要件（機能要件・非機能要件）
- 制約（期限・利用技術・既存仕様との整合性）
- 関連資料（仕様書・外部リンク）

曖昧な点が残る場合はユーザーに質問して解像度を上げてから次に進む。推測のまま設計に入ることを避けるため、このフェーズでの質問は積極的に行う。

### 2. 設計観点の動的決定

Issue内容から、本件で必要な観点を抽出する。以下のチェックリストから該当観点を選定し、不要な観点は省く、必要なら新規観点を追加する。

- **ドメイン**: エンティティ・値オブジェクト・リポジトリIFの追加／変更
- **アプリケーション**: ユースケース・ポートの設計
- **インフラストラクチャ**: Entスキーマ・外部APIクライアント・DBアクセス実装
- **プレゼンテーション**: HTTPハンドラ・ルーティング・リクエスト/レスポンスDTO
- **横断的関心事**: ロギング・設定・エラーハンドリング・認証認可・セキュリティ
- **テスト方針**: TDDで実装する際のテスト対象・テストデータ準備方針
- **インフラ環境**: Docker / AWS Lambda / ネットワーク・IAM構成

本プロジェクトはJSON API（フロントエンドを持たない）であるため、フロントエンド観点は通常含めない。Issue内容がCLI関連であればCLI観点を追加する等、プロジェクト特性に合わせて観点を柔軟に調整する。

選定した観点と「なぜその観点が必要か」をユーザーに提示し、観点の過不足を確認してから次に進む。

### 3. 並列設計の実行

観点ごとにサブエージェントを**1つのメッセージ内で並列起動**する。観点数分のサブエージェント呼び出しを同一メッセージに含めることで、順次実行ではなく並列実行される。

エージェントの使い分け方針:

- 既存コードの構造・責務を事前に把握する必要がある場合: `code-analyzer` サブエージェントを先行または並列で起動し、関連コードの現状を読み取り専用で調査させる
- 新規設計や方針決定: `general-purpose` サブエージェントを使用し、観点別に具体的な設計案（型定義・APIシグネチャ・レイヤー間依存・ファイル配置）を出力させる

各サブエージェントへのプロンプトには以下を含める。サブエージェントは親コンテキストの会話履歴を持たないため、自己完結した指示にすること。

- 対象Issueの要約
- 担当観点の明示（「あなたは〇〇観点のみを担当する」）
- プロジェクトの技術スタック（Go / DDD / Ent / PostgreSQL / Claude API / AWS Lambda想定）
- CLAUDE.mdの命名規則・interface適合イディオム・ディレクトリ構成・依存方向への準拠要求
- 期待する出力形式（例:「目的 / 設計案 / 影響範囲 / 懸念点」の節構成）

### 4. 成果物の統合とPlan提示

各サブエージェントの出力を読み、以下の観点で統合・調整する。

- **矛盾の解消**: レイヤー間でインターフェース型・命名・責務配置が食い違っていないか
- **依存方向の検証**: `presentation → application → domain ← infrastructure` の方向が守られているか
- **重複の排除**: 同じ責務が複数観点に現れていないか
- **抜け漏れの補完**: テスト・エラーハンドリング・ログ等の必須要素が揃っているか

統合したPlanを以下のテンプレートでユーザーに提示する。

```
# 設計Plan: Issue #<番号> <タイトル>

## 目的
<何を達成するか>

## 影響範囲
- 新規追加: <ファイル / 型 / パッケージ>
- 変更: <ファイル / 型 / パッケージ>

## 観点別設計
### <観点1>
<設計内容>

### <観点2>
<設計内容>

## 実装順序（TDDサイクル想定）
1. <コミット単位1: 最小機能または修正項目>
2. <コミット単位2>

## 懸念・未解決事項
- <設計時点で決めきれなかった点、ユーザー判断が必要な点>
```

実装順序は「コミット粒度は最低限の機能単位または修正項目一つずつ」（CLAUDE.md）に従い、1項目=1コミットの単位で列挙する。

### 5. ユーザー承認の取得

統合Planを提示し、ユーザーから明示的な承認を得る。承認が得られるまで次フェーズには進まない。

修正要求があった場合はフェーズ2〜4を必要な範囲でやり直す。特に「観点の抜け漏れ」「実装順序の見直し」はユーザーから出やすいため、該当箇所のみを再設計する。観点の大幅な追加が必要になった場合はフェーズ2から、設計内容のみの修正であればフェーズ4の統合作業から再開する。

### 6. 実装フェーズへの引き継ぎ

承認後、以下を通知して`/tdd-workflow`スキルへ引き継ぐ。

- Planの「実装順序」セクション
- 最初のコミット単位と、Redフェーズで書く失敗テストの対象
- 想定するAPIコンテナ内コマンド（例: `docker compose exec api go build ./...`）

## 原則

- **並列化を活かす**: 観点間に依存がない箇所は必ず並列サブエージェントで時短する。
- **CLAUDE.md規約を遵守**: 命名規則、interface適合イディオム、ディレクトリ配置、依存方向をすべての観点で意識する
- **成果物は日本語で出力**: Plan、設計案、コミット粒度の説明は日本語で記述する
- **Plan modeを外さない**: 承認前にファイル書き換えやコマンド実行による副作用を生じさせない
